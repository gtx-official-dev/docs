# Authentication

GTX DEX API provides both public and user-specific endpoints with different authentication requirements.

## Public Endpoints

Most GTX DEX API endpoints are public and do not require authentication:

- Market data endpoints (`/api/markets`, `/api/ticker/*`, `/api/depth`)
- Trading data endpoints (`/api/trades`, `/api/kline`)
- General information endpoints (`/api/pairs`)

<Note>
Public endpoints still have rate limits to ensure fair usage and system stability.
</Note>

## User-Specific Endpoints

User-specific endpoints require the user's wallet address as a parameter:

- `/api/account` - Account information and balances
- `/api/allOrders` - User order history
- `/api/openOrders` - User open orders
- `/api/cross-chain-deposits` - Cross-chain deposit history

### Address Parameter

User endpoints require a valid Ethereum wallet address:

```bash
curl "https://api.gtxdex.xyz/api/account?address=0x1234567890123456789012345678901234567890"
```

<Warning>
Wallet addresses must be valid Ethereum addresses (42 characters starting with "0x"). Invalid addresses will result in 400 Bad Request errors.
</Warning>

## Rate Limiting

GTX DEX API implements rate limiting to ensure fair usage:

### Rate Limit Headers

API responses include rate limit information in headers:

```http
X-RateLimit-Limit: 1200
X-RateLimit-Remaining: 1150
X-RateLimit-Reset: 1640995260
```

| Header | Description |
|--------|-------------|
| `X-RateLimit-Limit` | Maximum requests per window |
| `X-RateLimit-Remaining` | Requests remaining in current window |
| `X-RateLimit-Reset` | Timestamp when rate limit resets |

### Rate Limit Tiers

| Endpoint Category | Rate Limit | Window |
|-------------------|------------|--------|
| Market Data | 1200 req/min | 1 minute |
| Trading Data | 600 req/min | 1 minute |
| User Data | 300 req/min | 1 minute |
| Account Data | 200 req/min | 1 minute |

### Handling Rate Limits

When rate limits are exceeded, the API returns HTTP 429:

```json
{
  "error": "Rate limit exceeded",
  "retryAfter": 60
}
```

#### Best Practices

<Steps>
  <Step title="Implement Exponential Backoff">
    ```javascript
    async function makeRequest(url, retries = 3) {
      for (let i = 0; i < retries; i++) {
        try {
          const response = await fetch(url);
          if (response.status === 429) {
            const retryAfter = response.headers.get('Retry-After') || 60;
            await new Promise(resolve => setTimeout(resolve, retryAfter * 1000));
            continue;
          }
          return response;
        } catch (error) {
          if (i === retries - 1) throw error;
          await new Promise(resolve => setTimeout(resolve, Math.pow(2, i) * 1000));
        }
      }
    }
    ```
  </Step>

    ```javascript
      const reset = response.headers.get('X-RateLimit-Reset');
      
      if (remaining && parseInt(remaining) < 10) {

      }
    }
    ```
  </Step>
  
  <Step title="Cache Responses">
    ```javascript
    const cache = new Map();
    
    async function getCachedData(url, ttl = 5000) {
      const cached = cache.get(url);
      if (cached && Date.now() - cached.timestamp < ttl) {
        return cached.data;
      }
    ```

## Security Considerations

### Address Validation

Always validate wallet addresses on the client side:

```javascript
function isValidAddress(address) {
  return /^0x[a-fA-F0-9]{40}$/.test(address);
}

function getAccount(address) {
  if (!isValidAddress(address)) {
    throw new Error('Invalid wallet address');
  }
  
  return fetch(`https://api.gtxdex.xyz/api/account?address=${address}`);
}
```

### Data Privacy

<Warning>
User-specific endpoints return sensitive financial information. Always:
- Use HTTPS connections
- Validate address ownership on the client side
- Implement proper access controls in your application
- Never log or store sensitive user data unnecessarily
</Warning>

### Address Ownership Verification

For applications that modify user data, implement address ownership verification:

```javascript
// Example using ethers.js
async function verifyAddressOwnership(address, signature, message) {
  const recoveredAddress = ethers.utils.verifyMessage(message, signature);
  return recoveredAddress.toLowerCase() === address.toLowerCase();
}

// Usage in your application
const message = `Verify ownership of ${userAddress} at ${Date.now()}`;
const signature = await signer.signMessage(message);
const isOwner = await verifyAddressOwnership(userAddress, signature, message);

if (isOwner) {
  // Proceed with API call
  const account = await fetch(`/api/account?address=${userAddress}`);
}
```

## WebSocket Authentication

WebSocket connections currently do not require authentication but support user-specific subscriptions:

```javascript
const ws = new WebSocket('wss://ws.gtxdex.xyz');

// Subscribe to user-specific updates
ws.send(JSON.stringify({
  method: 'SUBSCRIBE',
  params: [`${userAddress}@account`, `${userAddress}@order`],
  id: 1
}));
```

<Note>
Future versions may implement WebSocket authentication for enhanced security.
</Note>

## Error Handling

Common authentication-related errors:

| Error Code | Description | Solution |
|------------|-------------|----------|
| 400 | Invalid address format | Validate address format |
| 429 | Rate limit exceeded | Implement backoff strategy |
| 500 | Server error | Retry with exponential backoff |

### Error Response Format

```json
{
  "error": "Address parameter is required",
  "code": "MISSING_ADDRESS",
  "timestamp": 1640995200000
}
```

## Production Considerations

When implementing authentication in production:

- **Secure Storage**: Never expose private keys in client-side code
- **Environment Variables**: Store sensitive data in environment variables
- **Rate Limiting**: Implement client-side rate limiting to respect API limits
- **Error Handling**: Properly handle authentication failures and retries
- **Key Rotation**: Regularly rotate API keys and update applications
